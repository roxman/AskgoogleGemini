# Ask Gemini Skill 使用指南（給任何 AI Agent）

透過操控使用者「本人已登入」的 Chrome，向 Google 的 AI 提問並擷取答案。不用 API（避免計費），不用 Playwright 無頭瀏覽器（Google 登入會被機器人偵測擋下）。成本為零：消耗的是使用者已付的訂閱額度（Google 帳號每日用量 + agent 自身的訂閱用量）。

## 部署到一台新電腦的前置條件

這份文件是自包含的：agent 讀完本文件即可執行，不需其他資料。但執行環境必須先滿足：

1. **Chrome 已安裝且已登入 Google 帳號**（Gemini 和 AI 模式都要登入才能用；這是人要先手動做的，agent 不能代登入）。
2. **Agent 要有操控該 Chrome 的能力**——任何能「導航到 URL」「在頁面上執行 JavaScript」「開／關分頁」的瀏覽器工具皆可：
   - **Claude in Chrome**（Cowork / Claude Code）：安裝 Claude in Chrome 擴充功能並登入。
   - **chrome-devtools-mcp**（Claude Code / Codex / Hermes 等）：`npx chrome-devtools-mcp@latest --browserUrl http://127.0.0.1:<CDP_PORT>`，需 Chrome 以偵錯模式啟動或由 MCP 附掛。
   - **Playwright MCP** 或其他瀏覽器 MCP：須附掛已登入 Google 的 Chrome profile。
   - **Hermes 內建 browser toolset**：在 profile config 設定 `browser.cdp_url` 即可。
   - 關鍵：必須操控「已登入 Google 的那個 Chrome profile」，空白或無痕實例沒登入、Gemini 用不了、Google 也容易擋。
3. **告知 agent 使用方式**：把本文件路徑（或內容）放進該 agent 的常駐指示（如 AGENTS.md、CLAUDE.md、SOUL.md、專案 instructions），註明「使用者明確要求問 Gemini 時，讀取並依照此文件執行」。

驗收方法：請 agent 導航到 `https://www.google.com/search?udm=50&q=test` 看能否讀到 AI 回答，再開 `https://gemini.google.com/app` 確認是登入狀態。兩項通過即可用。

## 觸發條件

只在使用者「明確」要求時使用：「問 Gemini」「請 Gemini 查」「用 Gemini 搜尋」等。未點名 Gemini 時，用 agent 自己的知識或內建搜尋。

## 模式一：快速模式（預設）— Google 搜尋 AI 模式

適合：單發查證、查資料。約 8–15 秒出結果，每次回答必有搜尋依據。

1. 把問題 URL encode，直接導航：
   `https://www.google.com/search?udm=50&q=<編碼後的問題>`
   問題自動送出，不需打字（因此沒有輸入吃字問題，數字可放心用）。
2. 等約 8 秒後讀取結果。**坑：此頁面用純文字擷取（如 innerText 類方法）常抓不到 AI 生成內容**，改用「執行 JS 擷取」或截圖讀取。還在生成就每 5 秒重試，上限 60 秒。
3. 回答右側有來源網站面板，一併記下。

## 模式二：對話模式 — gemini.google.com

適合：連續追問、長而複雜的多段問題。

1. 導航到 `https://gemini.google.com/app`。看到登入頁 → 停下請使用者手動登入，絕不代輸入帳密。
2. 定位輸入框：placeholder 類似「請輸入 Gemini 提示詞」，是 contenteditable 區塊（不是 textarea）。
3. 點擊後輸入問題。**實測的坑**：
   - **阿拉伯數字會被吃掉**（模擬輸入「2026年」只剩「年」）。數字改用中文數字（「二零二六年」），或輸入後截圖檢查，不完整就全選刪除重打。
   - **按 Enter 不會送出**：必須點輸入框右側的圓形送出箭頭按鈕。送出成功標誌：分頁標題變成對話主題、頁面出現「你說了」。
4. 等待：送出後 8 秒開始輪詢頁面文字，每 5 秒一次，直到連續兩次相同，或結尾出現「Gemini 是 AI，有時可能會出錯」（=生成完畢）。上限 90 秒。
5. 擷取：「Gemini 說了」之後到「Gemini 是 AI，有時可能會出錯」之前是回答本體。左側欄的歷史對話標題不是內容，要排除。

## 模式三：批次查證（量大、答案簡單的問題清單）

> ⚠️ **本節為早期版本，批次查證請直接用下方【最終正式版】甲法**（免打字＋JS 收割，已實戰驗證 553 筆）。以下保留原理說明（名稱清洗、多年份訊號判斷依據仍適用）。

適合：對一長串冷僻項目逐一查證（例如判斷比賽是歷屆還是一次性）。

1. **不要合併提問**：冷僻主題混在一起問容易張冠李戴。一題一分頁。
2. **先清洗比賽名稱**：去掉名稱中的年份（「2013」「115年」「2026年度」）與屆數（「第五屆」），只留純比賽名（「2013台北101商品設計比賽」→「台北101商品設計比賽」）。原理：搜尋純名稱時，SERP 本身就是判斷依據——歷屆比賽必然同時出現多個年份／屆數的結果，一次性活動只有單一年份；帶年份查會把結果鎖死在那一年。若清洗後太籠統（只剩「設計比賽」），保留主辦方等識別字。
3. 用快速模式平行開分頁，每批 3–5 個（太多會觸發 Google 限流），一起等結果再逐頁收割。
4. 提問固定模板，並要求 AI 利用多年份訊號：
   「『<清洗後的比賽名>』是每年都會辦理的歷屆比賽，還是一次性的活動？判斷依據：搜尋結果中是否同時出現多個不同年份或屆數。」
5. 收割完關閉分頁，換下一批。
6. 結果整理成清單回覆：項目｜歷屆比賽 or 一次性活動。

## 消化與回覆（共通原則）

1. 不要原文照貼。先用自己的知識檢查合理性，指出明顯錯誤或過時處；與自己判斷不同時並列兩邊觀點。
2. 分辨答案依據：有「真實可點的來源連結」＝有搜尋佐證；純文字宣稱來源但點不了＝可能只是模型記憶，要打折並標註「未經搜尋佐證」。
3. 用自己的話整理，標明哪些資訊來自 Gemini/AI 模式，附上來源連結。
4. 分頁留著不關（批次模式除外），讓使用者可自行查看原文或追問。

## 隱私與安全

- 提問內容會送到 Google：含私人檔案內容、金鑰、個資時，先徵求使用者同意或去識別化。
- 絕不代使用者輸入帳號密碼。
- 這是自動化操作消費者網頁：偶爾查資料沒問題，大量高頻使用可能觸發 Google 異常偵測，該規模應改用 API。

## 常見狀況

| 狀況 | 處理 |
|---|---|
| 歡迎彈窗／要求選模型 | 關掉或用預設，不糾結 |
| 回「我無法協助」 | 如實回報，改用自己的能力，不要反覆重試 |
| 達到用量上限 | 回報使用者（只是暫時限流，不會扣錢），改用自己的知識或搜尋 |
| 瀏覽器未連線 | 請使用者開啟 Chrome 確認擴充功能/偵錯連線，不要改走其他管道 |

## 省 token 模式（額度快用完時）

當 token 額度吃緊、且時間可犧牲（例如有排程每 15 分鐘接手）時，改用以下組合大幅降低單筆 token：

1. **去截圖**：收割只用文字搜尋或 JS 擷取（純文字擷取答案結論＋來源可點網址），不用截圖。實測文字擷取完整、無遺失；代價是 recurring 的逐屆明細會比截圖版粗（各屆對應網址不一定對齊），但 verdict／屆數／年份／來源等關鍵欄位齊全。
2. **解法 C — 單分頁循序（零失真、零風險）**：不要一次開多個分頁平行，改成「開 1 個分頁 → 等答案 → 收割 → 關掉 → 再開下一個」。系統每次工具回傳都會附「目前開啟分頁清單」，其中每個分頁都夾帶一條完整（且因把問題編碼進 q= 而超長）的網址；回音大小與「當下開著幾個分頁」成正比。只開 1 個分頁 → 每次呼叫只回音 1 條長網址，而非 5–10 條，token 大幅下降。問題原文完整送入、判斷不失真。代價：慢（循序等待，一批從約 5 分鐘拉到 10–12 分鐘），但排程可吸收此時間。

根因備忘：token 與「內容截斷需分兩次收割」的共同元凶，是那條把整段查證指令編碼進去的超長網址（一個中文字編碼成 9 字元，整段問題→上千字元），它在每次工具回傳的分頁清單裡被重複回音。縮短網址或減少同時開啟的分頁數，可同時緩解截斷與 token。

## 【最終正式版・實戰驗證 553 筆】甲法：短問題進網址 + 免打字 + JS 收割

批次查證的**確定主作法**（2026-07 用此法跑完 553 筆競賽歷屆查證，全清、零 needs_retry）。**取代下方解法 D**（D 需打字、較慢）。核心：問題編成短句、直接 URL 編碼進 `q=` 自動送出（免打字→全並行→快），收割用一段 JS 一次抓齊。

**問題模板（短句）**：
`『<清洗後比賽名>』是歷屆比賽嗎？它有其他年份屆次嗎？列出各屆年份與來源網址`
（名稱太籠統時前面補主辦關鍵字。數字放網址內，URL 編碼不吃數字。）

**流程（一批 N 筆，N≤10）**：
1. **開 N 個分頁**，每個分頁導航到 `https://www.google.com/search?udm=50&q=<URL編碼(短句問題)>`。若工具支援批次操作，盡量一次送出。免打字、自動送出。等約 12 秒讓 AI 生成。
2. **逐分頁執行收割 JS**：對每分頁執行一段 JavaScript，一次拿到「各屆年份＋結論句＋真來源 href」，並在最後用 replaceState 洗掉網址的 mstk。實測一支 JS 取代了多次頁面文字擷取與連結抽取，token 大降、且直接拿到可點網址。JS 範本見下。若工具支援批次執行 JS，盡量合併。
3. 收割完立刻關閉全部分頁。
4. 判讀、回填（在關分頁後做，不碰瀏覽器）。

**收割 JS 範本**（回傳精簡 JSON；末尾 replaceState 縮短後續分頁清單回音）：
```js
(() => {
  const seen=new Set(), links=[];
  document.querySelectorAll('a[href^="http"]').forEach(a=>{
    const h=a.href;
    if(/google\.|gstatic|schema\.org|policies|support\.|accounts\./.test(h))return;
    if(seen.has(h))return; seen.add(h);
    const t=(a.innerText||a.textContent||'').trim().replace(/\s+/g,' ').slice(0,45);
    if(t) links.push(t+' | '+h.split('?')[0].slice(0,110));
  });
  const bt=(document.body.innerText||'').replace(/\s+/g,' ');
  const years=[...new Set((bt.match(/20\d{2}/g)||[]))].sort();
  const sents=bt.split(/[。\n]/).map(s=>s.trim()).filter(s=>s.length>6);
  const concl=sents.filter(s=>/歷屆|系列|每年|一次性|僅有|沒有其他|其他年份|多個年份|屆/.test(s)).slice(0,4);
  history.replaceState({}, '', '/search?udm=50&q=x');   // 洗掉 mstk，縮短後續 tab-context 回音（實測有效）
  return JSON.stringify({years, concl, links: links.slice(0,20)});
})()
```

**收割備援**：JS 抓不到時（版面改版）退回逐段文字搜尋（先找結論＋年份、再找來源連結）。無論哪種都**禁用整頁純文字擷取**（AI 模式頁面常回空白或爆量 ~21k 字元）；只在補抽單一區塊時用定向文字擷取。

**跑這套的 agent 模型要求（重要，踩過大坑）**：需 **Sonnet 級以上**。實測小模型（Haiku 級）能做機械操作（讀寫檔、開分頁），但**無法完成「多分頁收割＋判讀」，會整批標 blocked**。**千萬別用 Fable**：它有獨立且較小的週額度，量大時先撞牆、且每輪消耗兇（實戰中是額度黑洞）。

**省 token 心法（根因公式）**：成本 ≈ 開著分頁數 × 每條網址長度 × 分頁開著期間工具呼叫次數。四個對策：問題短、猛併批（若工具支援批次，把多分頁的 JS 合併執行）、收割完秒關分頁、回填放最後。長網址的元兇是 Google 自加的 ~200 字元 mstk 追蹤碼，用上面 JS 裡的 replaceState 可洗掉。

**規模化實務**：每批 ≤10 分頁（>10 易觸發 Google 限流）；被 sorry／reCAPTCHA／限流的列 → 標 blocked、work_status=needs_retry、下輪自動重撿，不硬刷；一次多筆被擋就縮到 5 分頁。**避開尖峰時段**（消耗較快）。大量作業用排程每 N 分鐘接手一批，配鎖檔防重疊。

## 備選解法 D：短導航 + 打字送問題（舊法，已被甲法取代）

（歷史保留；甲法免打字更快更省。僅在「必須靠打字送複雜多段追問」時參考。）核心：網址只當「把比賽帶進 AI 模式」的短載具，真正的問題用打字送進輸入框——如此網址始終短（省 token、不截斷），問題卻完整送達（零失真）。

步驟（每筆）：
1. **短導航**：`https://www.google.com/search?udm=50&q=<比賽名稱+主辦關鍵字+年份>`。年份等阿拉伯數字**放在網址裡**（URL 編碼不會吃數字），不要放整段長指令。網址因此短，回音小。
2. 等約 8 秒 AI 生成初答（光名稱查詢常已顯示屆次線索）。
3. 找到底部「提出任何問題」輸入框 → 點擊 → **等 1–2 秒讓框醒過來** → 輸入「**零阿拉伯數字**」的框架追問，例如：「這個活動除了原始這一屆，還有沒有其他年份或屆次舉辦？請列出各屆年份、標題與可點來源網址。」（不含 0-9，就不會發生吃數字）。
4. 找到送出鈕（「傳送/送出」）並點擊（勿硬編座標，視窗大小會變）。等約 9 秒。
5. **收割**：用 JS 或文字搜尋抓「結論段落＋來源清單完整網址」，純文字、不截圖。
6. **fallback**：若只抓到來源連結、結論句太少，且比賽標題本身沒有屆次線索（如 vol.10＝第10屆、III＝第3屆），就再補一次定向文字擷取取結論句，再判。
7. 回填、關分頁、換下一筆。

為什麼數字不再被吃：吃數字只發生在「用 `type` 模擬打字、且從中文跳到阿拉伯數字的交界」。把數字全留在網址（走編碼、不走打字）、打字部分用零數字框架句，兩邊都避開了交界，數字 100% 保全。若不得已要打含數字的字，改成「分段送、在中文↔數字交界插 1 秒停頓」也可保住（實測有效）。

省 token 原理同前：導航網址只放短名稱 → 每次工具回傳的「分頁清單」裡那條網址短 → 長回音與內容截斷同時消失；問題走打字、送出後網址不被改寫（AI 模式追問不把問題寫回 URL），維持短。

## 【進階】大量作業的排程模式：心跳輪詢 ＋ FREE 旗標鎖（實戰 1400+ 筆驗證）

適用：一次要查數百上千筆、要**無人值守、跨數小時自動跑完**的清單。此模式與 ask-gemini 無關、是通用的排程骨架。實戰跑完三包（553＋484＋432＝1469 筆）驗證。

**架構四件套**：cron 心跳（polling）＋ 檔案互斥鎖（mutex/lease）＋ 逐列 done-check（防覆蓋）＋ 停止時間 guard。

### 為什麼是「輪詢」不是「事件驅動」
理想是事件驅動：一輪跑完 → 觸發下一輪（無縫、零浪費）。**但實測此環境的排程 run 沒有排程工具、無法自我重排、也無法自我停用**，所以「run 自串鏈」行不通。退而用**密集心跳輪詢**：讓 cron 一直回頭檢查，空就作業、忙就跳過。這就是 CS 的 polling watchdog / heartbeat。

### 心跳（cron）
用 **`*/2`（每 2 分鐘）** 讓 fire 夠密。一輪跑完後最多 2 分鐘就有下一拍接手。實測：心跳版換手間隔 ~1 分鐘、約每 6–7 分一輪；對比一般 `*/10` 因排程 jitter（dispatch 延遲數分鐘）+ 鎖門檻，會惡化成「每隔一發被跳過 → ~20 分一輪」。**心跳提速約 3 倍**。

### FREE 旗標鎖 ＋ 7 分租期救援（核心）
互斥鎖 = 確保同時只有一輪在跑（否則兩輪搶同一個瀏覽器、又同寫一個檔會打架）。鎖是一個檔 `.xxx_run.lock`，內容格式 `STATUS|ISO時間戳(+08:00)`，STATUS 為 `BUSY` 或 `FREE`。

進場判斷（每次心跳）：
```
讀鎖檔：
  不存在 / 空 / 非 BUSY 開頭(即 FREE 或壞格式)  → 空閒，進場
  BUSY 且 時間戳距今 < 7 分鐘                    → 上輪還在跑，跳過本輪
  BUSY 且 時間戳距今 ≥ 7 分鐘                    → 研判上輪掛了，進場接手
進場後立刻覆寫： BUSY|<現在>
本輪全部工作結束(存檔後)覆寫： FREE|<現在>   ← 這就是「解鎖」
```

**兩個關鍵設計，都是踩坑換來的**：

1. **用「寫 FREE」解鎖，不用「刪檔」解鎖。** 很多資料夾（如受管制的 staging）**不准刪檔**（`rm` 回 Operation not permitted）。若用「有檔案＝鎖著」的規則，第一輪建了鎖又刪不掉 → 永遠顯示鎖著 → 之後全部跳過 → **永久死鎖**。改用「覆寫成 FREE 旗標」當解鎖（覆寫被允許、刪除不被允許），乾淨又快。

2. **7 分鐘租期（TTL/lease）當防當機救援。** 若某輪跑到一半當掉（token 耗盡、瀏覽器斷線、遇 reCAPTCHA）就來不及寫 FREE，鎖會卡在 `BUSY` 又刪不掉。租期讓它「≥7 分自動視為過期」→ 下一拍照樣接手，**不會卡死**。門檻設「單輪時長 + 緩衝」：實測一輪 4–6 分 → 設 **7 分**最剛好。

> 名詞對照：這就是分散式系統的 **lease-based lock（帶租期的鎖）/ TTL lock**——正因為「無法保證乾淨釋放」才用時間自動過期，避免 deadlock。正常跑完＝FREE 立刻解鎖（快）；意外當掉＝租期自動救援（不死鎖）。

### 逐列 done-check、停止 guard、完成偵測
- 每輪只挑 `work_status` 為 `todo`／`needs_retry` 的列，**已 `done` 的一律跳過、不重寫**（冪等、防覆蓋）。
- 每輪先檢查「是否已過停止時間」，過了就寫 FREE、回報結束。
- 全部完成 → 回報「全部完成，請停用排程」且**不開瀏覽器**（只讀檔＋寫 FREE 就退出，近乎零成本）。**注意**：排程 run 無法自我停用，完成後 cron 仍會每 2 分鐘空跑一次直到停止時間或**有人手動關掉排程**。

### 搭配鐵則
模型 **Sonnet 級以上、絕不用 Fable**；每批 ≤10 分頁；限流退避；避開尖峰；鎖檔／timing log 放目標資料夾內；回填只寫 output 欄。

## 【進階】多題 funnel 模式：分層追問 ＋ 逐層關閉（實測驗證）

**廣度 vs 深度**：一般模式是「一題 × 大量」（廣度）。有時需要「同一項目連問 2–4 題、層層篩選」（深度）。深度**仍是通案**——對「當前子集」做通案分類，不是對單一項目問它專屬的問題（那就退化成人工用 Gemini）。例：Q1 這些哪些是動物 → Q2 是動物的裡哪些水生 → Q3 水生的裡哪些有肺。

### 何時啟用（重要，預設單題）
**預設一律單題模式**（甲法：一問、URL 送、平行、免打字，最快）。**只有下列情況才切多題 funnel**：(1) 使用者**明確說**有連續多題；(2) agent **開始前**就明確判定此任務需要分層。必須「開跑前就決定」，因為多題會影響：模式、鎖租期、時間估算、是否涉及打字。無法事先判定就用單題。

### 兩種深度做法，先選一種
- **複合一問（固定、單純、不需脈絡）**：把 2–4 個子問塞進一個問題模板、一趟問完、拆進多欄。最快、免打字、可完全平行。適合子問互相獨立、不需上一題脈絡時。
- **分層追問 funnel（需脈絡連續、或只想深挖觸發子集）**：見下。

### 分層追問 funnel 流程（實測通過）
一批 N 分頁：
1. **Q1**：URL 送出（`udm=50`，平行、免打字），等 12s，逐分頁**執行 JS 收割**。
2. **逐層關閉**：依 Q1 答案，**當場關閉不觸發下一題的分頁**（例：非動物關掉）；**觸發的留著**。早關 = 後續開著的分頁少 = 回音省 token。
3. **Q2 打字追問**：對留著的分頁，定位底部「提出任何問題」輸入框 → 點擊 → 輸入「**零阿拉伯數字**」框架追問（避開吃數字）→ 送出（AI 模式按 Enter 有效）。**對話脈絡會保留**：問「牠…」Gemini 知道「牠」＝上一輪主體（實測「牠」跨三輪都正確指向鯨魚）。
4. 收割 Q2 → 再逐層關閉 → Q3 → …
5. **一輪結束條件**：所有題問完 ＋ 所有分頁關閉。

### 收割關鍵（多輪專用）
多輪對話答案**往下疊加**，最新答案在**頁面最尾端**。所以**執行 JS 抓 `body 尾端`**（或「最後一個『以下是你所說的內容：<問題>』之後的文字」）。**別用「全頁關鍵字過濾」**——會抓到上一輪的舊答案（實測踩過）。

### 成本與設定
- 打字是**逐分頁序列**（不能同時打多框）；每層要等 ~15–20s 生成；**一輪時間變長且不固定**。
- 配排程時：**心跳鎖租期要調大**（單題一輪 4–6 分設 7 分；多題三層一輪可能 12–15 分 → 租期設 ~18–20 分），停止時間估算也要放寬。
- 仍守：每批 ≤10 分頁、限流退避、Sonnet 級模型、避尖峰。

### 不在範疇
**自適應追問**（必須先看到 Q1 答案、才能想出 Q2 要問什麼）不自動化——因為那意味著中間要「人去分析、去想」，自然就是「先跑完這批 → 人看結果 → 想出新模板 → 再跑一趟」，每趟仍是現有批次，skill 不需為此新增東西。
